# Onda 1: logger manual e store SQL

## Objetivo

Entregar o nucleo executavel de auditoria manual da issue #9 sobre o bootstrap
da PR #20. Uma chamada fluente cria uma `Activity`, a persiste por SQL
parametrizado e a leitura pela store a devolve com o mesmo conteudo. A entrega
inclui SQLite, PostgreSQL e MySQL, schema/migrations de referencia e testes de
paridade com os exemplos de manual logging do Spatie.

## Limites

Esta onda implementa a escrita e a leitura direta de `ActivityStore`. A Query
API fluente, cursores e filtros JSON pertencem a #13; por enquanto a store
aceita os filtros estruturados necessarios para observar atividades e rejeita
filtros de `properties` com uma excecao de capacidade explicita. `prune()`
sera implementado por SQL, sem CLI (que continua na fase operacional).

O core nao importa TypeORM, Prisma, Drizzle ou um driver de banco. O unico
driver adicionado e `better-sqlite3`, em `devDependencies`, para o teste de
integracao SQLite em memoria. PostgreSQL e MySQL usam os servicos existentes
no `docker-compose.yml` e executam apenas quando `PG_HOST` e `MYSQL_HOST`
estiverem definidos.

## API publica

```ts
const logger = createActivityLogger({ store, logName: 'default' });

await logger
  .activity('billing')
  .performedOn(subjectRef('Order', order.id))
  .causedBy(causerRef('User', userId))
  .withProperties({ plan: 'pro' })
  .event('subscribed')
  .log('Subscription created');
```

`createActivityLogger` cria uma instancia sem estado global. `ActivityLogger`
expoe `activity(logName?)`; portanto cada configuracao de store conserva seu
proprio default e o uso continua explicitamente seguro em processos com mais
de um banco. O builder e imutavel: cada metodo devolve um novo builder e nao
vaza estado para a proxima atividade.

O builder fornece `performedOn`/`on`, `causedBy`/`by`,
`causedByAnonymous`/`byAnonymous`, `withProperties`, `createdAt`, `event`,
`tap` e `log`. Sem `createdAt`, `log` chama `createActivityTimestamp`; a data
e UTC, clonada e com precisao de milissegundos conforme D19. `withProperties`
faz merge raso de propriedades. `tap` pode enriquecer os campos conhecidos e
`properties`; campos SQL arbitrarios nao existem no contrato TypeScript e nao
sao persistidos. Esta e a adaptacao segura do exemplo Spatie de `tap`.

`log` cria uma `NewActivity` com `subject`, `causer`, `event` e `batchUuid`
nulos quando omitidos, persiste uma unica linha e devolve `Promise<void>`.
O contrato congelado de `ActivityStore.persist` e `void`; a atividade e lida
de volta por `store.query`, que tambem e a costura de observacao dos testes.

## Store SQL

`SqlExecutorStore` recebe uma `SqlDataSource`:

```ts
interface SqlDataSource {
  dialect: 'sqlite' | 'postgres' | 'mysql';
  execute(sql: string, params?: readonly unknown[]): Promise<readonly unknown[]>;
}
```

Ela implementa `ActivityStore`. Em `persist(activities, transaction)`, usa
`transaction` quando fornecida e o executor normal nos demais casos. Assim um
adapter de ORM pode passar o executor preso a transacao viva e conservar o
invariante iff-committed; a store nao tenta abrir, commitar ou reverter uma
transacao por conta propria.

Os valores sao sempre parametros posicionais. `tableName` passa por
`assertSafeIdentifier` antes de ser interpolado e cada parte de um nome
qualificado e escapada pelo dialeto. O nome padrao e `activity_log`.

O mapeamento de colunas e fixo: `id`, `log_name`, `description`,
`subject_type`, `subject_id`, `causer_type`, `causer_id`, `event`,
`properties`, `batch_uuid` e `created_at`. Identificadores de subject/causer
sao armazenados como string, `properties` como JSON e `created_at` como ISO
UTC com milissegundos. A leitura reconstrui `Date`, valores nulos e JSON para
o tipo `Activity`.

`query` implementa os filtros estruturados de `ActivityFilter` que nao exigem
semantica JSON, com ordenacao estavel por `created_at, id`. `properties` e
cursor ficam para #13; a solicitacao de filtro JSON falha com erro prefixado
`activitylog:` em vez de prometer suporte parcial. `prune` remove por
`created_at`, opcionalmente limitado por `log_name`, e retorna a contagem
afetada quando o executor a disponibilizar.

## Dialetos e migrations

O core tera estrategias pequenas para placeholder e quoting: `$1..$n` e
aspas duplas no PostgreSQL, `?` e crases no MySQL, `?` e aspas duplas no
SQLite. As migrations de referencia terao:

- PostgreSQL: `BIGINT GENERATED BY DEFAULT AS IDENTITY`, `jsonb` e
  `timestamptz(3)`.
- MySQL: `BIGINT UNSIGNED AUTO_INCREMENT`, `JSON` e `datetime(3)` em UTC.
- SQLite: `INTEGER PRIMARY KEY AUTOINCREMENT`, `TEXT` JSON e ISO UTC em
  `TEXT`.

Cada migration inclui o indice de `log_name` e os indices compostos de
subject e causer. Snippets equivalentes de TypeORM, Prisma e Drizzle ficam
como referencia no diretorio de migrations, sem importar esses pacotes no
core.

## Erros

As excecoes publicas estendem `Error`, definem `name` e produzem mensagens que
comecam com `activitylog:`. Esta onda introduz erro de identificador invalido,
erro de data invalida e erro de capacidade para filtro nao suportado.

## Testes e verificacao

`test/logger.spec.ts` usa SQLite real `:memory:` e reproduz cada exemplo de
manual logging do Spatie: descricao simples, subject e alias, causer e alias,
causer anonimo, propriedades, `createdAt`, event e `tap`. Cada caso observa a
linha por `store.query`, nao por mocks.

Testes adicionais verificam merge imutavel do builder, UTC/milissegundos,
uso do executor transacional, serializacao JSON, identificador malicioso,
erros prefixados, `prune` e a estrutura SQL de todos os dialetos. A matriz
PostgreSQL/MySQL cria a tabela pela migration correspondente e roda os mesmos
cenarios quando os servicos Docker estiverem disponiveis.

Antes de publicar a atualizacao da PR: `npm audit`, lint, typecheck, testes,
build, dry-run de publish e a sequencia equivalente via `docker-compose`
(OrbStack). A revisao final compara o diff com `origin/main` e com os
criterios da issue #9.

- Preserva D13: valores vinculados; identificadores validados e escapados.
- Preserva D19: timestamp nasce na aplicacao e a ordem e `created_at + id`.
