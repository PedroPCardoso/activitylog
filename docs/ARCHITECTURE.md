# ARCHITECTURE — decisões (estilo grilling)

Tabela viva de decisões arquiteturais, cada uma resultado de uma pergunta grelhada. Ver `CONTEXT.md` para o glossário e invariantes de domínio.

| # | Decisão | Alternativa rejeitada | Por quê |
|---|---|---|---|
| D1 | **iff-committed** é o invariante definidor: uma Activity persiste se e somente se a mutação commitar, *quando a mutação for transacional*. Fora de tx → best-effort. | Sync-in-tx como mero "nice-to-have", tolerando órfãos. | Trilha de auditoria confiável vale mais que throughput. Ver `CONTEXT.md` › Invariants. |
| D2 | **Core agnóstico de ORM + adapters específicos.** A store só precisa de `(dialect, SqlExecutor)`; o `TransactionRef` passado em `persist()` é um executor SQL ligado à tx viva do ORM. Uma única `SqlExecutorStore`, reusando dialects/`assertSafeIdentifier` do metrics também na escrita. | Uma store nativa por ORM (Caminho B), escrevendo via API do ORM. | Requisito: funcionar independente do ORM. Adapter específico é inevitável (só ele alcança o handle de tx), mas o core permanece limpo e uma store só. |
| D17 | **MVP 0.1 ancorado no TypeORM; Prisma fast-follow 0.2.** TypeORM demonstra iff-committed impecável (`event.databaseEntity` + `event.manager`) e valida toda a pilha com o mínimo de ressalvas; Prisma (iff via `auditedTransaction`, `$extends` best-effort) entra logo depois para capturar o gap antes do ZenStack. | Prisma-first (aposta de mercado do plano). | A estreia deve ser "audit trail que nunca deixa órfão", não "com asteriscos"; TypeORM tem o menor risco de execução para isso. |
| D16 | **Ordem do pipeline de escrita: `capturar diff → beforePersist(hook) → redaction → persist`.** Redaction é o guardião final; o que o hook adicionar também é redigido. Hook **não** desliga redaction; gravar campo sensível de propósito exige allowlist explícita, nunca bypass. `dontSubmitEmptyLogs` avaliado após redaction (pula tarde). | `beforePersist` rodar depois da redaction / poder desligá-la. | Um hook que enriquece `properties` reintroduziria PII/segredo já limpo; redaction por último fecha o vetor (A09). |
| D15 | **Exatamente 3 pacotes, base neutra `activitylog-`; suporte a ORM funcional DENTRO dos pacotes de framework — nunca pacote por ORM.** `activitylog-core` (agnóstico Nest/Hono/Fastify), `activitylog-nestjs` (integração Nest independente de ORM — módulo/DI/contexto/interceptor funcionam com qualquer store; **TypeORM como subpath interno**), `activitylog-nextjs` (**Prisma + Drizzle como subpaths internos**). | Pacote por ORM (`activitylog-prisma`/`-typeorm`/`-drizzle`) — explicitamente rejeitado. | Consistência com o irmão metrics e simplicidade > SEO por ORM. **Mitigar** a perda de descoberta via `keywords` npm e READMEs por adapter. |
| D14 | **Redaction deny-by-default (A09):** lista de nomes sensíveis embutida e ligada por default (sobreponível/desativável), match por nome, **deep**, case-insensitive; valor → `'[REDACTED]'` (chave preservada). Redige no DiffEngine antes de persistir e antes de qualquer log. Fail secure (mascara/descarta no erro, nunca persiste cru). | Opt-in puro (usuário esquece um campo → vaza). | Audit log de um `User` capturaria hash de senha/PII/token por padrão; deny-by-default é a postura fail-secure da org e um diferencial real. |
| D13 | **Choke point de injeção (A05):** `tableName` e qualquer nome de coluna customizável passam por `assertSafeIdentifier` antes do SQL; `subject_type`/`causer_type`/`event`/`log_name`/`subject_id`/`properties` são sempre bind params. Spec de segurança prova rejeição de `tableName` malicioso. **Fronteira:** uma `store` custom (implementação externa de `ActivityStore`) é responsabilidade de segurança do usuário. | Interpolar `tableName` sem validação; confiar que valores string são "seguros". | OWASP A05: identificadores não parametrizáveis são o único vetor; um choke point único é auditável. |
| D12 | **Outbox → 1.0 (fora do 0.x); `ActivityStore` único (sem split writer/reader); assinatura congelada com `persist(activities, ctx?: TransactionRef)`.** Outbox é implementação futura da mesma interface (persist→outbox, query→tabela final); `ctx` habilita sync-in-tx hoje e outbox ignora amanhã. | Outbox no 0.x; interface já dividida writer/reader. | MVP não carrega complexidade não validada; split writer/reader é abstração especulativa (regra do três). |
| D11 | **`properties` = melhor tipo consultável por dialeto:** `jsonb` (PG), `JSON` (MySQL), `TEXT` (SQLite). Serialização canônica (chaves reordenadas), não literal. | `json` literal no PG. | `properties` é diff estruturado, não documento byte-a-byte; consulta/índice habilita os scopes da query API. |
| D10 | **PK `bigint` default, `uuid`(v7) configurável.** Sync-in-tx numa store SQL única (caso comum) → banco atribui bigint no INSERT sem custo; paridade Spatie. Outbox/distribuído → opta por uuid v7 gerado no cliente. `batch_uuid` é uuid independentemente. | `uuid` v7 default desde o 0.x. | Não impor custo de índice uuid a todo mundo por um caso avançado; default serve os 99%, config destrava o resto. |
| D9 | **Agregado = 1 Activity** com `subject_id=null`, `subject_type`=model, `properties={aggregate:true, criteria, changes, affected}`, sem `old`. Nunca expandir bulk em N linhas por default. Query API ganha scope pra incluir/excluir agregados. | Tipo `*Bulk` separado; expandir per-row quando `affected` pequeno. | Nunca fingir diff por linha; explosão de linhas e falso diff destroem confiança. |
| D8 | **Old-values sem lock por default (ideia do Spatie), janela documentada, lock opt-in.** "old" = linha visível na tx imediatamente antes do UPDATE (ver `CONTEXT.md` › Old). `lockForDiff: true` para exatidão sob concorrência. Teste da §9 demonstra a janela. | `SELECT ... FOR UPDATE` por default. | Spatie lê o `original` em memória sem lock; default sem lock evita contenção/deadlock e serve 99% dos apps. |
| D7 | **Fronteira de fila = contrato explícito.** Helpers `serializeContext()` (enfileirador) + `runWithContext(serialized, fn)` (worker) para propagar causer/batch pelo payload do job. Default quando não há contexto reestabelecido: **causer nulo (system)** — nunca adivinhar, nunca vazar. iff-committed continua valendo na tx do job. | Só nota de limitação na doc, sem API. | ALS não cruza a fronteira de enfileiramento; sem API o boilerplate todo cai no usuário e o causer some sem aviso. |
| D6 | **Abrir a ALS cedo, resolver o causer tarde.** A store da ALS é aberta no **middleware** (propagação robusta), guardando uma referência à request. O causer é resolvido **preguiçosamente no `.log()`**, quando o guard já rodou e `request.user` existe. Interceptor é alternativa secundária. | Popular o causer no interceptor (snapshot eager). | Guard roda depois do middleware e antes do interceptor; ALS via `Observable` do RxJS é frágil. Lazy resolve mata os dois problemas. |
| D5 | **Contexto via ALS singleton no core.** Uma única instância de `AsyncLocalStorage` exportada do `nestjs-activitylog-core`, carregando causer/batch/withoutLogging. O core é **dep compartilhada e deduplicada** (`external` no tsup, nunca bundled nos leaves) — restrição de empacotamento mais forte que a do metrics, porque o estado não pode ser duplicado. nestjs-cls só como receita de interop, sem acoplamento. | Ler contexto direto do nestjs-cls quando presente. | Estado duplicado em memória → causer some silenciosamente entre adapters (duas ALS). Metrics podia ignorar (stateless); activitylog não. |
| D4 | **Identidade de Subject/Causer:** `*_type` = string opaca (default nome da classe, sobreponível por morph-map); `*_id` = `varchar` universal, guardando a PK stringificada (bigint/uuid/string). Tipagem forte só no TS (`subjectRef<T>`). PK composta fora de escopo no v0. | Amarrar `subject_id` ao tipo real da PK (JOIN/índice tipado). | Core não conhece classes nem ORMs; JOIN de `subject_id` é raro numa trilha de auditoria, então universalidade > tipagem da coluna. |
| D3 | **iff-committed nos adapters só é garantido pelo helper explícito de transação.** Decorator (`@LogsActivity` TypeORM) e `$extends` (Prisma) são **conveniência best-effort com lacunas documentadas**; os caminhos iff são `auditedUpdate` (TypeORM) e `auditedTransaction` (Prisma). Uma **matriz de cobertura** declara, por adapter × operação, onde vale iff e onde é best-effort. | Perseguir iff automático no `$extends`/subscriber. | Prisma `$extends` não entrega o tx client para writes irmãos; subscriber TypeORM não dispara em `.update()`/QB/bulk (typeorm#4537). Honestidade sobre cobertura é diferencial, não fraqueza. |
